# cantor-node

The node owns a stable Ed25519 identity, claims its relay room, authenticates
apps end to end, and serves the Phase 2 capability/status protocol. Relay
disconnects are retried forever with capped, jittered exponential backoff.

Start the relay from the repository root in one terminal:

```sh
cd relay
npm install
npm run dev
```

Then start one-time pairing in another terminal:

```sh
cd node
cargo run -p cantor-node -- pair
```

Scan the terminal QR from Cantor, or paste the printed `cantor://pair` URI. The
app sends a key-bound HMAC proof rather than the raw pairing token. The token is
invalidated after the first client proves possession of its Ed25519 key; that
public key is written to `allowed_keys` in `node.toml`. Subsequent launches use
`run` and accept only keys already in that allowlist.

The first run creates `node.key` and `node.toml` under the platform config
directory (`~/.config/cantor` on Linux). Both files are owner-only, and the key
contains the raw 32-byte Ed25519 secret. The default relay is
`ws://localhost:8787`.

For an isolated first run or a non-default node name:

```sh
cargo run -p cantor-node -- pair \
  --config-dir /tmp/cantor-local-node \
  --name cesar-desktop \
  --relay-url ws://localhost:8787
```

The first-run overrides are rejected once `node.toml` exists; edit that file for
later configuration changes. Never copy or commit `node.key`.

For a command-line handshake/status demo, keep `pair` running and pass its URI
to the script (quote the URI so the shell does not interpret `&`):

```sh
node scripts/protocol-client.mjs 'cantor://pair?...' \
  --identity /tmp/cantor-demo-client.json
```

Run the same command again to exercise the persisted allowlist. Add
`--omit-token` with a fresh identity path to verify that an unlisted key gets
the application-level `rejected` error.

`cargo test --workspace` also regenerates the committed TypeScript definitions
under `../protocol`; those files are generated by `ts-rs` and should not be
edited manually.

## Linux installer stub

`install.sh` installs a release binary into `~/.local/bin`, writes a new
owner-only config under `~/.config/cantor`, and creates a hardened per-user
systemd unit. It deliberately does not start the unit before an app has paired.
The default production relay is `wss://cantor.ckadirt.xyz`.

The release path downloads the architecture-specific GitHub asset and verifies
its adjacent `.sha256` file before installation:

```sh
curl -fsSLO https://raw.githubusercontent.com/ckadirt/Cantor/main/node/install.sh
less install.sh
sh install.sh
```

Until release assets are published, build locally and exercise the same
installer path explicitly:

```sh
cargo build --release -p cantor-node
CANTOR_NODE_BINARY="$PWD/target/release/cantor-node" \
  CANTOR_NODE_NAME=cesar-desktop \
  ./install.sh
```

Existing `node.toml` files are preserved on reinstall. The installer refuses
symlinked config/service targets and unmanaged `cantor-node.service` files.
After installation, use the printed `cantor-node pair` command, scan its QR,
then stop the foreground process and enable the service:

```sh
systemctl --user enable --now cantor-node.service
systemctl --user status cantor-node.service
```
